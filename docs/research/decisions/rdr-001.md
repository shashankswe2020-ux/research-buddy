# RDR-001: Select Query Complexity Classification Approach

## Status
Accepted

## Date
2026-04-19

## Context
AdaptRAG requires a mechanism to classify query complexity into discrete tiers.
The classifier drives the entire adaptive retrieval strategy, making it a critical
component. Key requirements:
- Must be fast (<50ms) to avoid bottlenecking the pipeline
- Must achieve ≥85% accuracy for adaptation to be beneficial
- Must generalize across query domains (not just QA datasets)

## Decision
Fine-tune DeBERTa-v3-base with a 3-class classification head, trained on 10K
queries derived from existing QA datasets.

## Alternatives Considered

### Use the LLM itself (zero-shot prompting)
- Pros: No additional model needed; can leverage LLM's understanding
- Cons: 200-500ms latency per classification; inconsistent output format; uses
  generation budget for a classification task
- Rejected: Too slow and unreliable for a pipeline component

### Rule-based classifier (keyword patterns + query length)
- Pros: Zero latency; fully interpretable; no training needed
- Cons: Brittle; 65% accuracy in initial tests; fails on paraphrased queries
- Rejected: Accuracy too low — would degrade adaptation below random tier assignment

### Binary classification (simple vs. complex)
- Pros: Simpler; higher per-class accuracy
- Cons: Loses the intermediate tier, which handles 36% of queries; forces
  k=2 or k=10 with no middle ground
- Rejected: Three tiers provide better coverage of the complexity spectrum

## Consequences
- DeBERTa-v3-base adds ~440MB to deployment size
- Requires 10K labeled training examples (one-time cost)
- 30ms inference overhead per query (acceptable)
- Achieves 87.3% accuracy — above the 70% threshold where adaptation breaks even
